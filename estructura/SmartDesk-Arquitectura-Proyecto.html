01 · Visión del Producto
¿Qué es SmartDesk?
Plataforma SaaS B2B de gestión de servicios internos con triaje automático vía IA
🏢
Para empresas B2B
Entornos aislados por empresa (Tenant). Cada organización tiene su propia instancia segura con datos completamente separados.
🤖
Triaje automático con IA
Google Gemini clasifica, prioriza y asigna tickets automáticamente al área correcta en menos de 5 segundos.
🆓
Modelo Freemium
Sandbox gratuito para probar el producto. Licencia B2B para empresas formales con SSO, CSV masivo y soporte avanzado.
🎭 Roles del sistema
  • 👑
    Administrador de Empresa — gestiona el entorno, colaboradores, áreas y configuración corporativa
  • 🔧
    Técnico / Resolutor — atiende tickets, los toma, comenta y cierra con propuesta de solución
  • 👤
    Cliente / Colaborador — crea tickets, sigue el estado y valida la resolución
📊 Métricas clave del sistema
  • ⏱Triaje IA ≤ 5 segundos por ticket
  • ⚡Navegación y consultas ≤ 2 segundos bajo carga normal
  • 🔒Disponibilidad mínima 99.5% mensual (24/7)
  • 📱Interfaz responsiva, español, formatos dd/mm/aaaa
  • 🛡RBAC estricto + segregación lógica multi-tenant

04 · Multi-Tenancy
Aislamiento por Esquema PostgreSQL
Cada empresa tiene su propio esquema aislado en la misma base de datos
📦 Esquema public — Global
Compartido

Contiene los datos globales del sistema, independientes de cualquier empresa:

  • 📄
    tenants — registro de empresas, plan_type, subdominio
  • 👤
    usuarios_globales — email, password, is_active
  • 🔑
    verification_tokens — OTP, tokens de activación y recuperación
  • 📋
    user_tenant_roles — relación usuario ↔ tenant ↔ rol
🏢 Esquema tenant_[nombre] — Por Empresa
Aislado

Creado automáticamente al activarse cada empresa. Contiene:

  • 🗂
    areas — departamentos de la empresa (TI, RRHH, etc.)
  • 👥
    usuarios — colaboradores del tenant con sus roles locales
  • 🎫
    tickets — incidencias con estado, prioridad, IA metadata
  • 💬
    ticket_mensajes — comentarios público/privado e historial
  • 🔔
    notificaciones — bandeja in-app por usuario
  • 📊
    audit_log — accesos, IPs y cambios de estado
🔄 Cómo funciona el routing multi-tenant
1
Request HTTP
acme.smartdesk.com
2
JWT Filter
Extrae tenant_id del token
3
TenantContext
Set search_path = tenant_acme
4
JPA Repository
Queries auto-aisladas al esquema
5
Response
Solo datos de ese tenant

07 · Seguridad
Autenticación y Control de Acceso
Spring Security + JWT + OTP por correo electrónico
🔑
JWT (JSON Web Tokens)
  • ✓Emitido tras login exitoso
  • ✓Contiene user_id, tenant_id, role
  • ✓Expira en 60 minutos de inactividad
  • ✓Revocado en logout y suspensión
  • ✓Validado en cada ruta protegida
📧
OTP por correo (MFA)
  • ✓Registro: token de activación único
  • ✓Recuperación de contraseña: token 1 uso
  • ✓Expiración de 24h (configurable)
  • ✓Invalidación tras uso
🛡
Spring Security + RBAC
  • ✓RBAC: ADMIN, TECNICO, CLIENTE
  • ✓Bloqueo tras 5 intentos fallidos (15 min)
  • ✓HTTPS obligatorio en toda comunicación
  • ✓BCrypt para hash de contraseñas
  • ✓Audit log de todos los accesos
Flujo de autenticación completo
// 1. Usuario envía credenciales al login POST /api/v1/auth/login { "email": "user@acme.com", "password": "••••••••" } // 2. Spring Security valida BCrypt + tenant + estado cuenta // 3. Si OK → JWT emitido con claims del tenant { "sub": "user-uuid", "tenant_id": "acme-uuid", "role": "TECNICO", "schema": "tenant_acme", "exp": 1750000000 } // 4. Cada request usa el token → TenantFilter setea search_path SET search_path TO tenant_acme, public;

08 · Inteligencia Artificial
Triaje Automático con Google Gemini
LLM procesa cada ticket y devuelve clasificación estructurada en < 5 segundos
📥 Input al LLM
{ "descripcion_usuario": "No puedo acceder a mi correo corporativo desde ayer", "areas_disponibles": ["TI", "RRHH", "Facturación"], "contexto_empresa": "Empresa de logística, 50 empleados" }

{ "titulo": "Acceso bloqueado a correo corporativo", "area": "TI", "prioridad": "Alta", "keywords": ["email", "acceso", "bloqueado"], "razonamiento": "Incidencia de acceso afecta productividad diaria", "solucion_sugerida": "Revisar configuración MX y estado de cuenta en AD" }

09 · Notificaciones
Sistema de Alertas en Tiempo Real
WebSockets para app abierta + FCM para app cerrada
⚡
WebSockets (STOMP)
Cuando el usuario tiene la app abierta. Notificación instantánea en el ícono de campana.
  • ✓Nuevo comentario en ticket
  • ✓Cambio de estado del ticket
  • ✓Ticket crítico o urgente asignado al área
  • ✓Propuesta de resolución recibida
🔥
Firebase FCM (Push)
Cuando el técnico tiene la app móvil cerrada o en segundo plano.
  • ✓Ticket crítico asignado al área
  • ✓Ticket en proceso con SLA excedido
  • ✓Solo para tickets Alta/Crítica
📨
Correo Transaccional
Para eventos clave que requieren registro formal fuera de la plataforma.
  • ✓Token de activación (registro)
  • ✓Recuperación de contraseña
  • ✓Invitación de colaborador
  • ✓Boletín de SLA excedido (admin)
  • ✓Propuesta de cierre al creador

🎫 Ciclo de vida de un Ticket con IA
1
Descripción
Colaborador redacta el problema
2
Gemini IA
Triaje: área, prioridad, título, keywords
3
Ticket ABIERTO
Persiste en BD con metadata de IA
4
Técnico toma
Autoasignación → EN PROCESO
5
Propone resolución
Técnico envía solución al cliente
6
Cierre y rating
Cliente valida → calificación con estrellas
📊 Estados de un Ticket
ABIERTO Ticket creado, esperando técnico disponible
ASIGNADO Técnico tomó el ticket de la cola
EN PROCESO Técnico trabajando activamente en el problema
RESUELTO Cliente validó la solución propuesta
CERRADO Ticket cerrado. Sistema solicita calificación
🚦 Prioridades de Tickets
🔴 CRÍTICA Afecta operación completa. Notificación a toda el área
🟠 ALTA Impacto significativo en productividad de una persona
🟡 MEDIA Funciona con limitaciones, puede esperar
🟢 BAJA Mejora o consulta. Sin urgencia operativa
